ted displays git blame annotations as ghost text directly in the editor — move your cursor to any line to see who last changed it and when. The annotation appears at the end of the line in faint, unobtrusive text, giving you authorship context without interrupting your reading or editing flow.

## How it works

The blame feature watches your cursor position. Each time you rest on a new line for a moment, ted looks up the blame information for that line and renders the result as inline ghost text at the end of the current line. The lookup is non-blocking and triggered automatically — no command or shortcut required.

## What the annotation shows

Each blame annotation contains two pieces of information:

| Field | Example | Description |
|---|---|---|
| **Author** | `Maya Patel` | Name of the committer as recorded in git |
| **Date** | `2 days ago` | Relative time since the commit was made |

The annotation reads like: `Maya Patel, 2 days ago`, rendered as ghost text after the last character on the line.

## No configuration needed

Blame is active by default for every file that belongs to a git repository. You do not need to enable it in settings or run any command. If the file is not tracked by git, the annotation is simply absent.

<Note>
  Blame annotations are read-only and decorative — they do not affect the file content, the editor's undo history, or any git state.
</Note>

## Combining blame with Commit History

Blame and Commit History are designed to work together. When an annotation points to a change you want to understand more deeply, open the Commit History panel to trace the full context of that commit.

<Steps>
  <Step title="Spot an interesting annotation">
    Move your cursor to a line whose blame annotation — author or date — you want to investigate further.
  </Step>
  <Step title="Open Commit History">
    Click the clock icon in the sidebar to open the Commit History panel.
  </Step>
  <Step title="Locate the commit">
    Find the matching commit in the graph by author and date. Hover over it to expand the full diff summary, see every file the commit touched, and read the original commit message.
  </Step>
</Steps>

<Tip>
  This workflow is especially useful for understanding *why* a line exists: the commit message and surrounding diff often contain the rationale that isn't visible in the code itself.
</Tip>

## Annotation updates as you move

The blame annotation always reflects the line where the cursor currently sits. As you navigate through a file — whether by clicking or using arrow keys — the annotation updates automatically to show the blame for the new current line. Moving to a new line clears the previous annotation until the lookup for the new line completes.
