ted uses [web-tree-sitter](https://github.com/tree-sitter/tree-sitter/tree/master/lib/binding_web) for syntax highlighting instead of the regex-based approaches common in many editors. Tree-sitter parses your source file into a full Abstract Syntax Tree (AST) on every keystroke — incrementally, so only the changed portions are re-parsed — delivering accurate, context-aware highlighting even for complex nested structures like JSX inside TypeScript, or deeply templated Rust generics. For languages where a tree-sitter grammar is not bundled, ted falls back to the battle-tested lezer grammars that ship with CodeMirror 6.

## How It Works

Parsing is computationally intensive enough that running it on the main UI thread would cause noticeable lag. ted offloads all tree-sitter work to a dedicated **Web Worker** that runs in parallel with the rest of the editor.

<Steps>
  <Step title="File opens in the editor">
    ted detects the language from the file's extension and instructs the worker to load the matching tree-sitter grammar (a compiled `.wasm` binary).
  </Step>
  <Step title="Worker parses the AST">
    The worker parses the full file on first load, then re-parses only the changed ranges as you edit. Parsing never blocks the UI thread.
  </Step>
  <Step title="Highlighting ranges are sent to the main thread">
    The worker posts token ranges and their semantic types back to the main thread. CodeMirror applies the corresponding theme classes to those ranges in the editor.
  </Step>
  <Step title="Display updates instantly">
    Because the worker runs concurrently, the editor remains fully responsive while the AST is being built or rebuilt.
  </Step>
</Steps>

<Info>
  The incremental re-parsing strategy means that a large file with a small edit only re-parses the region around the change — not the entire file. This keeps highlighting latency imperceptible even in files with thousands of lines.
</Info>

## Supported Languages

ted ships highlighting for the following languages out of the box. Languages marked **Tree-sitter** use compiled `.wasm` grammar bundles running in the background worker; languages marked **Lezer** use CodeMirror's built-in grammar engine.

| Language | File Extensions | Highlighter |
|---|---|---|
| JavaScript | `.js`, `.mjs`, `.cjs`, `.jsx` | Tree-sitter |
| TypeScript | `.ts`, `.tsx` | Tree-sitter |
| Python | `.py` | Tree-sitter |
| Rust | `.rs` | Tree-sitter |
| C / C++ | `.c`, `.cpp`, `.h`, `.hpp`, `.cc` | Tree-sitter |
| HTML | `.html`, `.htm` | Tree-sitter |
| CSS | `.css`, `.scss`, `.less` | Tree-sitter |
| JSON | `.json` | Tree-sitter |
| Java | `.java` | Lezer |
| XML | `.xml`, `.svg` | Lezer |
| YAML | `.yml`, `.yaml`, `.toml` | Lezer |
| Markdown | `.md`, `.mdx`, `.markdown` | Lezer |

## Language Detection

ted determines the language for a file automatically by matching its extension against the table above. No configuration is required — open a `.rs` file and Rust highlighting is active immediately.

<Note>
  If a file has no extension or uses an uncommon extension, ted falls back to plain-text mode with no highlighting. You can override this manually by selecting a language from the status bar at the bottom of the editor pane.
</Note>

## Theming

Syntax colors are driven by ted's semantic theming system. Each token type (keyword, string, comment, function name, and so on) maps to a semantic color role defined by the active theme.

To switch themes, open **Settings → Appearance → Theme** and select from the available options. The change takes effect immediately across all open editor tabs.

<Tip>
  Because theming is semantic rather than hard-coded per language, installing a new theme automatically applies correct colors to every language ted supports — you never need a separate theme extension per language.
</Tip>

## Beyond Highlighting

The AST produced by tree-sitter grammars serves double duty in ted. In addition to powering syntax highlighting, it feeds the **AI agent** with structural context about your code — giving the agent a precise understanding of function boundaries, variable scopes, and code structure before it proposes edits or suggests refactors.
